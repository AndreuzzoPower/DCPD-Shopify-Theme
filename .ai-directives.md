## 2. Standard Obbligatori

### Accessibilità
- Tutto il codice DEVE essere conforme a **WCAG 2.1 AA**
- Usare attributi ARIA appropriati (`aria-label`, `aria-current`, `aria-hidden`, `role`, ecc.)
- Garantire navigazione da tastiera e focus visibili
- Usare `<span class="visually-hidden">` per testi accessibili non visivi

### Gerarchia Heading
- **`<h1>`**: UNICO per pagina, riservato al titolo principale della pagina
- **`<h2>` e superiori**: NON usare tag heading per elementi decorativi o non semantici. Usare `<div>` con la classe corrispondente (es. `class="h2"`)
- Se un elemento ha già una classe, mantenerla; se non ne ha, aggiungere `class="hN"` corrispondente

### Dati Strutturati (Rich Snippets)
- I file contengono già rich snippet (dati strutturati JSON-LD)
- Se una modifica impatta i dati strutturati esistenti → **aggiornarli**
- Se nuove funzionalità permettono dati strutturati specifici → **crearli**
- Formati: `application/ld+json` con schema.org

## 3. Convenzioni Codice

### Liquid
- Usare `{%- -%}` (con trattino) per eliminare whitespace dove possibile
- Variabili con nome descrittivo in `snake_case`
- Commenti con `{%- comment -%}` per sezioni importanti
- **Tutte le label e descrizioni nello schema DEVONO essere in italiano** — NON usare chiavi di traduzione `t:...` per le sezioni custom MS. Le chiavi `t:` sono riservate alle sezioni Dawn originali che hanno traduzioni complete in tutti i file locales.

### CSS
- **NON** scrivere CSS inline nelle sezioni. Creare sempre un file CSS dedicato in `assets/` (es. `ms_nome-sezione.css`)
- Caricare il CSS nella sezione con `{{ 'ms_nome-sezione.css' | asset_url | stylesheet_tag }}`
- Usare `#section-{{ section.id }}` per specificità e isolamento
- `!important` solo quando necessario per sovrascrivere Dawn
- Mobile-first: stili base → media query `min-width: 750px` per desktop
- `box-sizing: border-box` su tutti gli elementi custom
- Eccezione: stili dinamici che dipendono da settings Liquid possono usare `{%- style -%}` inline

### JavaScript
- Vanilla JS (NO jQuery)
- Event delegation quando possibile
- Nessun `console.log` in produzione

### Shopify Schema
- Organizzare i settings con `header` per raggruppamento logico
- Usare `default` per tutti i settings
- Usare tipi appropriati: `select`, `checkbox`, `range`, `color`, `richtext`, `image_picker`
- **NON usare mai emoji o icone** nei campi `"content"` degli `header` dello schema. Utilizzare solo testo semplice (es. `"content": "Layout sezione"`, NON `"content": "📐 Layout sezione"`)
- **Range: il valore `default` DEVE essere un multiplo esatto di `step`** — es. con `step: 5` i valori validi sono 0, 5, 10, 15... Non usare valori come 36 o 28 con step 5. Shopify rifiuta il salvataggio del file con errore `FileSaveError`.
- **Range: numero step ≤ 100** — `(max - min) / step` DEVE risultare ≤ 100. Es. max:1600, step:10 → 160 step → INVALIDO. Usare step:20 → 80 step → valido. Shopify rifiuta il file con errore di schema validation.
- **Select: NON usare stringa vuota come `default`** — Shopify può rifiutare silenziosamente il deploy di un intero file se un setting `select` ha `"default": ""`. Usare SEMPRE un valore sentinella non vuoto come `"inherit"`, `"none"`, `"auto"`, ecc. e gestirlo nel Liquid con `if setting != 'inherit'`. Esempio corretto:
  ```json
  { "value": "inherit", "label": "Eredita da desktop" }   ← opzione "nessuna scelta"
  ```
  ```liquid
  {%- if section.settings.my_select != 'inherit' and section.settings.my_select != blank -%}
    ... applicare il valore ...
  {%- endif -%}
  ```

### Padding `.page-width` su mobile/tablet nelle sezioni con carousel

Alcune sezioni Dawn rimuovono il padding del contenitore `.page-width` su mobile/tablet per permettere al carousel di scorrere a bordo schermo. Questo causa la perdita del padding anche quando il carousel è disabilitato.

**Pattern Dawn noto:**
- `section-multicolumn.css`: `.multicolumn .page-width { padding: 0 }` su `≤989px`
- `section-collection-list.css`: `.section-collection-list .page-width { padding: 0 }` su `≤749px`

**Regola:** Ogni sezione MS che:
1. Ha un outer div con una classe corrispondente a un selettore Dawn che rimuove padding
2. Usa un setting `section_width`

DEVE includere un blocco `{% style %}` che ripristina il padding corretto quando il carousel non è attivo:

```liquid
{%- unless show_carousel or sw == 'full_width' -%}
  {% style %}
    @media screen and (max-width: 767px) {
      #section-{{ section.id }} .page-width {
        padding-left: 1.5rem;
        padding-right: 1.5rem;
      }
    }
    @media screen and (min-width: 768px) and (max-width: 989px) {
      #section-{{ section.id }} .page-width {
        {%- if sw == 'narrow' -%}
        padding-left: 9rem;
        padding-right: 9rem;
        {%- else -%}
        padding-left: 5rem;
        padding-right: 5rem;
        {%- endif -%}
      }
    }
  {% endstyle %}
{%- endunless -%}
```

**Sezioni MS attualmente affette:** `ms_multicolumn.liquid` (risolto). Tutte le altre sezioni MS non hanno questo problema perché l'outer div non ha la classe target dei selettori Dawn.

## 7. Standard per Creazione / Modifica Section

### 7.1 Naming Convention
- Ogni section DEVE essere nominata con prefisso `ms_` → es. `ms_nome-sezione.liquid`
- **Eccezione**: i file con prefisso `main-` (es. `main-product.liquid`, `main-collection-product-grid.liquid`) mantengono il nome originale perché sono referenziati direttamente dai template JSON di Shopify
- Se si rinomina una section esistente, verificare **tutti i riferimenti** (template JSON, snippet, layout) e aggiornarli
- Il nome nello schema (`"name"`) deve iniziare con `MS` → es. `"name": "MS Custom HTML/LIQUID"`
- **Title Case obbligatorio**: ogni parola nel nome della sezione DEVE avere la prima lettera maiuscola → es. `"MS Featured Product"`, `"MS Rich Text"`, `"MS Blog In Evidenza"`. NON `"MS Featured product"` o `"MS Rich text"`.

### 7.2 Categorie Section (Presets)
Ogni section deve avere un `"category"` nei preset, scegliendo tra queste categorie predefinite:

| Categoria | Quando usarla |
|---|---|
| **Contenuti** | Sezioni di testo, HTML custom, contenuti editoriali |
| **Media** | Immagini, video, gallerie |
| **Developer** | Sezioni tecniche, codice custom, utilità |
| **Beta** | Funzionalità non ancora completate o in fase di test |
| **Navigazione** | Menu, breadcrumb, sidebar di navigazione |
| **Prodotti & Collezioni** | Griglie prodotti, banner collezione, filtri |

> ⚠️ Se una nuova section non rientra in nessuna categoria, **chiedere sempre** all'utente prima di crearne una nuova. Se approvata, aggiornare questa lista.

### 7.3 Blocchi Obbligatori: Padding, Margin, Background
Ogni section DEVE includere i blocchi speciali per la gestione di padding, margin e background.

**Riferimento**: `sections/ms_custom-html.liquid`

I blocchi da includere:
- **`section_padding`** — Range sliders per padding top/bottom su Desktop, Tablet, Mobile (limit: 1)
- **`section_margin`** — Range sliders per margin top/bottom su Desktop, Tablet, Mobile (limit: 1)
- **`section_background`** — Image picker per immagine BG + color picker per colore BG (limit: 1)

I blocchi generano stili tramite `{%- render 'ms_section-style' -%}` (incluso dopo la chiusura del contenitore principale).

### 7.4 Tag Heading Personalizzabile
Quando un elemento ha un **title** o **subtitle**, aggiungere SEMPRE la possibilità di scegliere il tag HTML:

```json
{
  "type": "select",
  "id": "title_tag",
  "label": "Tag titolo",
  "options": [
    { "label": "H1", "value": "h1" },
    { "label": "H2", "value": "h2" },
    { "label": "H3", "value": "h3" },
    { "label": "H4", "value": "h4" },
    { "label": "H5", "value": "h5" },
    { "label": "H6", "value": "h6" },
    { "label": "Paragrafo", "value": "p" },
    { "label": "Div", "value": "div" }
  ],
  "default": "h2"
}
```

E la possibilità di aggiungere una **classe CSS personalizzata**:

```json
{
  "type": "text",
  "id": "title_custom_class",
  "label": "Classe CSS personalizzata titolo",
  "info": "Classe aggiuntiva (opzionale). Le classi esistenti vengono mantenute."
}
```

**Nel Liquid** il rendering sarà:
```liquid
{%- assign title_tag = section.settings.title_tag | default: 'h2' -%}
{%- assign title_classes = 'classe-base-esistente' -%}
{%- if section.settings.title_custom_class != blank -%}
  {%- assign title_classes = title_classes | append: ' ' | append: section.settings.title_custom_class -%}
{%- endif -%}
<{{ title_tag }} class="{{ title_classes }}">{{ section.settings.title }}</{{ title_tag }}>
```

> ⚠️ **Eccezione**: Se il campo è di tipo `liquid` o `html`, il tag heading non è necessario perché l'utente può inserire tag manualmente.

> ⚠️ **Modifica di moduli esistenti**: Se l'elemento ha già delle classi, **mantenerle sempre**. La classe custom si aggiunge in coda.

### 7.4.1 Dimensione Visiva Heading (`heading_size` / `title_size`)

Quando un heading ha un setting per la **dimensione visiva** (classe CSS indipendente dal tag HTML), usare SEMPRE le classi Dawn standard. Le classi `h3`, `h4`, `h5`, `h6` NON esistono come classi di sizing in Dawn — corrispondono solo ai tag HTML nativi.

**Variante standard (sezioni e blocchi principali):**

Usare questa in tutte le section e i block, salvo contesti speciali (footer, accordion).

```json
{
  "type": "select",
  "id": "heading_size",
  "label": "Dimensione titolo",
  "options": [
    { "label": "Piccola", "value": "h2" },
    { "label": "Media", "value": "h1" },
    { "label": "Grande", "value": "h0" },
    { "label": "Extra grande", "value": "hxl" },
    { "label": "Extra extra grande", "value": "hxxl" }
  ],
  "default": "h1"
}
```

| Classe CSS | Label | Dimensione approssimativa |
|---|---|---|
| `h2` | Piccola | ~1.6rem |
| `h1` | Media | ~2.4rem |
| `h0` | Grande | ~3.2rem |
| `hxl` | Extra grande | ~4.8rem |
| `hxxl` | Extra extra grande | ~6.4rem |

**Variante footer (contesti piccoli):**

Solo per blocchi del footer dove le dimensioni standard sono troppo grandi.

```json
{
  "options": [
    { "label": "Piccola", "value": "h6" },
    { "label": "Media", "value": "h5" },
    { "label": "Grande", "value": "h4" },
    { "label": "Extra grande", "value": "h3" }
  ],
  "default": "h5"
}
```

**Variante accordion (contesti collapsible):**

Solo per `ms_collapsible-content.liquid` e simili.

```json
{
  "options": [
    { "label": "Extra piccola", "value": "h6" },
    { "label": "Piccola", "value": "h5" },
    { "label": "Media", "value": "h4" },
    { "label": "Grande", "value": "h3" },
    { "label": "Extra grande", "value": "h2" }
  ],
  "default": "h4"
}
```

**Rendering Liquid standard:**

Il tag HTML e la classe di dimensione sono **indipendenti**: il tag controlla la semantica, la classe controlla l'aspetto visivo.

```liquid
{%- assign title_tag = section.settings.title_tag | default: 'h2' -%}
{%- assign title_classes = section.settings.heading_size | default: 'h1' -%}
{%- if section.settings.title_custom_class != blank -%}
  {%- assign title_classes = title_classes | append: ' ' | append: section.settings.title_custom_class -%}
{%- endif -%}
<{{ title_tag }} class="{{ title_classes }}">{{ section.settings.title }}</{{ title_tag }}>
```

**Checklist rapida:**
- [ ] La dimensione visiva usa le classi Dawn (`h2`, `h1`, `h0`, `hxl`, `hxxl`)?
- [ ] NON usa classi inesistenti (`h3`, `h4`, `h5`, `h6`) come classi di sizing per contesti standard?
- [ ] Le label sono descrittive ("Piccola", "Media", "Grande") e NON tecniche ("H2", "H1")?
- [ ] Il default è coerente con la variante scelta (`h1` per standard, `h5` per footer, `h4` per accordion)?

### 7.5 Section Width Obbligatoria
Ogni section DEVE includere il setting **Section Width** con queste opzioni:

```json
{
  "type": "select",
  "id": "section_width",
  "label": "Larghezza blocco",
  "options": [
    { "label": "Stretta (narrow)", "value": "narrow" },
    { "label": "Larghezza pagina", "value": "page_width" },
    { "label": "Larghezza fluida", "value": "page_fluid" },
    { "label": "Piena larghezza", "value": "full_width" }
  ],
  "default": "page_width"
}
```

**Opzioni disponibili**:
| Valore | Classe CSS (Dawn) | Classe CSS (Bootstrap) | Comportamento |
|---|---|---|---|
| `narrow` | `page-width page-width--narrow` | `container` | Larghezza stretta (controllata da `settings.page_width_narrow`, default 750px) |
| `page_width` | `page-width` | `container` | Larghezza pagina (controllata da `settings.page_width`, default 1200px) |
| `page_fluid` | `page-width page-width--fluid` | `container-fluid` | 100% larghezza **con** padding laterale (nessun max-width) |
| `full_width` | *(nessuna classe)* | *(nessuna classe)* | Piena larghezza del viewport, senza padding |

**CSS**: `.page-width--fluid` è definita in `assets/base.css` e semplicemente rimuove `max-width` da `.page-width`, mantenendo il padding.

**Setting globale**: Il valore di `page-width--narrow` è controllato dalla variabile CSS `--page-width-narrow`, definita in `layout/theme.liquid` e configurabile dall'utente nei **Settings > Layout > Larghezza contenuto stretto** (`settings.page_width_narrow`).

**A livello di blocco**: Quando una section ha blocchi con layout indipendente (es. `main-article.liquid`), il `section_width` va aggiunto **a ogni singolo blocco** con default appropriato (`narrow` per contenuti testuali, `full_width` per immagini hero, ecc.).

**Retrocompatibilità**: Se la section aveva valori precedenti (`default`, `contained`, `full`, `fluid`), mapparli ai nuovi:
| Vecchio valore | Nuovo valore |
|---|---|
| `default` / `contained` | `page_width` |
| `full` | `full_width` |
| `fluid` | `page_fluid` |

Aggiungere nel Liquid:
```liquid
if sw == 'default' or sw == 'contained'
  assign sw = 'page_width'
elsif sw == 'full'
  assign sw = 'full_width'
elsif sw == 'fluid'
  assign sw = 'page_fluid'
endif
```

**Pattern Liquid per calcolo classe (Dawn)**:
```liquid
{%- liquid
  assign sw = section.settings.section_width | default: 'page_width'
  if sw == 'default'
    assign sw = 'page_width'
  endif
  case sw
    when 'narrow'
      assign sw_class = 'page-width page-width--narrow'
    when 'page_width'
      assign sw_class = 'page-width'
    when 'page_fluid'
      assign sw_class = 'page-width page-width--fluid'
    when 'full_width'
      assign sw_class = ''
  endcase
-%}
```

**Pattern Liquid per calcolo classe (Bootstrap)**:
```liquid
{%- liquid
  assign sw = section.settings.section_width | default: 'page_width'
  if sw == 'default'
    assign sw = 'page_width'
  endif
  case sw
    when 'narrow' or 'page_width'
      assign sw_bs_class = 'container'
    when 'page_fluid'
      assign sw_bs_class = 'container-fluid'
    when 'full_width'
      assign sw_bs_class = ''
  endcase
-%}
```

**Riferimento implementazione**: `sections/main-article.liquid` (Dawn), `sections/ms_custom-html.liquid` (Bootstrap)

### 7.6.1 Carosello Desktop quando esiste Carosello Mobile
> **Regola**: Se una section supporta la visualizzazione a carosello (slider) su mobile, DEVE supportarla anche su desktop, con le stesse opzioni di configurazione.

**Pattern obbligatorio per sezioni con carosello:**

| Setting | ID | Sezione schema | Descrizione |
|---|---|---|---|
| Colonne desktop | `columns_desktop` | Desktop | Numero colonne in griglia desktop |
| Abilita carosello desktop | `enable_desktop_slider` | Desktop | Attiva/disattiva slider su desktop |
| Autoplay desktop | `enable_autoplay` | Desktop | Abilita rotazione automatica |
| Velocità autoplay | `autoplay_speed` | Desktop | Secondi tra le slide (range 1-10) |
| Colonne mobile | `columns_mobile` | Mobile | Numero colonne in griglia mobile |
| Abilita swipe mobile | `swipe_on_mobile` | Mobile | Attiva/disattiva slider su mobile |

**Logica Liquid standard:**
```liquid
assign show_mobile_slider = false
if section.settings.swipe_on_mobile and items_count > columns_mobile_int
  assign show_mobile_slider = true
endif

assign show_desktop_slider = false
if section.settings.enable_desktop_slider and items_count > section.settings.columns_desktop
  assign show_desktop_slider = true
endif
```

**Classi CSS slider:**
- `slider-component-desktop` su `<slider-component>` quando desktop slider è attivo
- `slider--desktop` su `<ul>` quando desktop slider è attivo
- `slider--tablet grid--peek` su `<ul>` quando mobile slider è attivo
- Slider buttons: `large-up-hide` se desktop slider è OFF, `medium-hide small-hide` se mobile slider è OFF

**Data attributes autoplay:**
```html
<slider-component data-autoplay="{{ section.settings.enable_autoplay }}" data-speed="{{ section.settings.autoplay_speed }}000">
```

### 7.8 Settings Globali e Override a Livello di Section

> **Principio**: quando si creano impostazioni che hanno senso come default globale (es. dimensione pulsante, stile tipografico, colori di base), queste vanno inserite in `config/settings_schema.json`. Tutti gli elementi del tema che rientrano in queste impostazioni devono applicarle come **default**. Se una section specifica ha lo stesso parametro nel suo schema, quel valore **sovrascrive** il default globale.

**Flusso di priorità**:
1. Valore nella section/block → **priorità massima** (override)
2. Valore globale in `settings_schema.json` → **default** (se nella section è impostato "Predefinita" o blank)
3. Valore hardcoded nel CSS/Liquid → **fallback** (se tutto il resto è vuoto)

**Esempio pratico — Dimensione pulsante**:
- **Globale**: `settings.ms_buttons_default_size` in `config/settings_schema.json` (sezione "Pulsanti")
- **Section**: setting `button_size` con opzione "Predefinita" (value: `"default"`)
- **Snippet helper**: `snippets/ms_button_size.liquid` gestisce il fallback automaticamente
- **CSS**: `assets/ms_button-sizes.css` definisce le classi `.button--size-{small,medium,large,jumbo,extra-jumbo}`

**Quando creare un nuovo setting globale**:
- Se il parametro si applica a 3+ sections diverse
- Se ha senso che l'utente lo imposti una sola volta per tutto il sito
- Se il valore per section è un "override" piuttosto che una configurazione specifica

**5 taglie standard pulsanti**: small, medium (Dawn default), large, jumbo, extra-jumbo.

**Snippet `ms_button_icon.liquid`** — renderizza l'icona nella posizione corretta:
- `slot: 'before'` → icona prima del testo (se `position == 'before'`)
- `slot: 'after'` → icona dopo il testo (se `position == 'after'`)
- Se `icon` è vuoto, non renderizza nulla.

**Snippet `ms_button_classes.liquid`** — genera le classi:
- `type: 'button'` → `button [button--outline] button--size-*`
- `type: 'pill'` → `button button--pill [button--outline] button--size-*`
- `type: 'link'` → `link underlined-link`

`button--outline` viene aggiunto solo quando `secondary` è `true`.
Per `type: 'link'` il parametro `secondary` viene ignorato (nessun bordo/sfondo).

**Logica contorno + forma:**
- Il bordo segue **sempre** la forma del pulsante: rettangolare per `button`, arrotondato per `pill`.
  Questo perché `.button--outline::after` eredita il `border-radius` del parent (default Dawn) o quello di `.button--pill::after` (`50px`).

**CSS:** definiti in `assets/ms_button-sizes.css`:
- `.button--pill` — `border-radius: 50px` su elemento, `::before` e `::after`.
- `.button--outline` — `--alpha-button-background: 0` (sfondo trasparente), `--border-opacity: 1` su `::after` (bordo visibile), testo che eredita `--color-button`.

**Per pulsanti multipli** (es. button_1 / button_2), aggiungere il suffisso `_1`, `_2` a tutti gli ID:
`button_size_1`, `button_style_secondary_1`, `button_type_1`, `button_icon_1`, `button_icon_position_1`, ecc.

### 7.11 Sistema Minimo d'Ordine per Prodotto

> **Principio**: è possibile impostare una quantità minima d'ordine per ogni prodotto. La funzionalità è configurabile globalmente e sovrascrivibile per singolo prodotto tramite metafield.

#### Settings globali

In `config/settings_schema.json`, gruppo **"Minimo d'ordine"**:

| Setting | Tipo | ID | Default |
|---|---|---|---|
| Attiva minimo d'ordine | `checkbox` | `ms_min_order_enabled` | `false` |
| Quantità minima predefinita | `number` | `ms_min_order_default` | `1` |

#### Metafield prodotto

`custom.minimo_d_ordine` — valore numerico. Se > 0, sovrascrive il default globale per quel prodotto.

#### Logica di calcolo (inline, no snippet separato)

```liquid
{%- liquid
  assign ms_min_qty = 0
  if settings.ms_min_order_enabled
    assign meta_val = product.metafields.custom.minimo_d_ordine.value | default: 0 | plus: 0
    if meta_val > 0
      assign ms_min_qty = meta_val
    elsif settings.ms_min_order_default > 0
      assign ms_min_qty = settings.ms_min_order_default | plus: 0
    endif
  endif
  assign native_min = product.selected_or_first_available_variant.quantity_rule.min | default: 1
  if ms_min_qty > native_min
    assign effective_min = ms_min_qty
  else
    assign effective_min = native_min
  endif
-%}
```

Il valore finale (`effective_min`) è il **maggiore** tra il minimo nativo Shopify e il nostro minimo custom, per non sovrascrivere mai una regola Shopify più restrittiva.

#### Punti di applicazione

La logica inline va applicata in **tutti** i punti dove un prodotto viene visualizzato con possibilità di acquisto:

1. **Pagine prodotto**: `main-product.liquid`, `ms_featured-product.liquid` — override `min`/`data-min`/`value` sull'input quantity + `data-ms-min-order` per validazione JS
2. **Card prodotto**: `card-product.liquid` — logica inline + testo "Min: X" + `data-ms-min-order` sul form quick-add
3. **Snippet quantity**: `quantity-input.liquid` accetta parametro `ms_min_order` + override `data-min`/`data-ms-min-order`
4. **Quick order list**: `quick-order-list-row.liquid` — calcola minimo e lo passa a `quantity-input`
5. **Buy buttons**: `buy-buttons.liquid` — `data-ms-min-order` su `<product-form>`
6. **Carrello drawer**: `cart-drawer.liquid` — override `data-min` + warning sotto item + checkout button disabilitato se violazione
7. **Carrello pagina**: `main-cart-items.liquid` — override `data-min` + warning
8. **Carrello footer**: `main-cart-footer.liquid` — checkout button disabilitato se violazione

#### Testo informativo

Mostrare "Quantità minima d'ordine: X" **solo se `effective_min > 1`**. Usare la chiave localizzata:

```liquid
{{ 'products.product.quantity.min_order' | t: quantity: effective_min }}
```

Chiavi in `locales/`:
- `products.product.quantity.min_order` — messaggio informativo
- `products.product.quantity.min_order_cart_warning` — warning nel carrello

#### Validazione a 3 livelli

1. **HTML**: attributo `min` sull'input quantity
2. **JavaScript**: `product-form.js` valida pre-submit leggendo `data-ms-min-order`; `cart.js` valida in `validateQuantity` leggendo `data-ms-min-order`
3. **Liquid server-side**: il carrello ricalcola il minimo server-side ad ogni render; se violato, il bottone checkout viene disabilitato

> **Nota**: un utente che chiama direttamente `/checkout` via URL può aggirare tutto. Per bloccare anche questo serve **Shopify Functions** (Cart Validation, disponibile con Shopify Plus o app).

### 7.12 Personalizzazione Carosello per Sezione

> **Principio**: ogni sezione che utilizza un carosello (`slider-component`) deve esporre un set standard di 8 settings per la personalizzazione del comportamento e dell'aspetto del carosello, gestiti interamente a livello di sezione.

**Ordine obbligatorio dei settings nel gruppo "Carosello"** (dopo `enable_desktop_slider`/`columns_desktop`, prima di "Mobile"):

1. `enable_autoplay` / `enable_desktop_slider_autoplay` — checkbox
2. `autoplay_speed` / `desktop_slider_autoplay_velocita` — range
3. **`carousel_loop`** — select ← subito dopo autoplay
4. **`carousel_peek_size`** — select ← subito dopo loop
5. `carousel_arrows_position` — select
6. `carousel_arrows_overlay_dark` — checkbox
7. `carousel_pagination_style` — select
8. `carousel_scroll_mode` — select
9. `carousel_show_arrows` — checkbox
10. `carousel_show_pagination` — checkbox

| ID | Tipo | Default | Opzioni | Note |
|---|---|---|---|---|
| `carousel_loop` | select | `circular` | `none`, `rewind`, `circular` | `none`=nessun loop, `rewind`=torna al primo visibile, `circular`=loop infinito senza rewind |
| `carousel_peek_size` | select | `medium` | `none`, `small`, `medium`, `large` | — |
| `carousel_arrows_position` | select | `below_center` | `below_center`, `overlay_sides`, `above_right` | — |
| `carousel_arrows_overlay_dark` | checkbox | `false` | — | Solo per `overlay_sides` |
| `carousel_pagination_style` | select | `counter` | `counter`, `dots`, `numbers`, `none` | — |
| `carousel_scroll_mode` | select | `page` | `page`, `single` | — |
| `carousel_show_arrows` | checkbox | `true` | — | — |
| `carousel_show_pagination` | checkbox | `true` | — | — |

**Regole loop:**
- `carousel_loop` default = `"circular"` (NON più checkbox, NON più `false`)
- Se `carousel_loop` è `"none"`, autoplay si ferma all'ultima slide e il bottone passa in stato play; cliccando play il carosello riparte dalla slide 1
- Backward compat: i vecchi valori `"true"`/`"false"` salvati nei template JSON vengono mappati automaticamente a `rewind`/`circular`

**Snippet condiviso**: `snippets/ms_carousel-controls.liquid`
Parametri: `section_id`, `total_items`, `pagination_style`, `show_arrows`, `show_pagination`, `enable_autoplay`.
NON duplicare l'HTML delle frecce e della paginazione nelle sezioni: usare sempre questo snippet.

### 7.12.1 Embla Carousel — Architettura e Regole

> **Solo `ms_slideshow.liquid`** usa **Embla Carousel v9.0.0-rc01**, abilitato dall'attributo `data-ms-embla="true"` su `<slideshow-component>`.
> I caroselli generici (`<slider-component>`) usano il motore nativo basato su `scroll-snap` + `SliderComponent` in `global.js`. NON applicare Embla ai caroselli generici: il CSS grid (colonne, peek, gap) è incompatibile con l'approccio `transform: translate3d()` di Embla.

#### File e caricamento

| File | Ruolo | Caricamento |
|---|---|---|
| `assets/embla-carousel.umd.js` | Core Embla v9.0.0-rc01 | `layout/theme.liquid`, **prima** di `global.js`, con `defer` |
| `assets/embla-carousel-accessibility.umd.js` | Plugin ARIA v9.0.0-rc01 | `layout/theme.liquid`, **prima** di `global.js`, con `defer` |
| `assets/global.js` | `SliderComponent` + `SlideshowComponent` | `layout/theme.liquid` con `defer` |

**CRITICO — Compatibilità versioni:** Core e plugin accessibility **devono** essere la stessa major version. La v8 usa API `scrollNext/scrollTo/scrollSnapList`, la v9 usa `goToNext/goTo/snapList`. Un mismatch causa `TypeError: a.snapList is not a function` e blocca l'inizializzazione.

#### Ordine di caricamento in `layout/theme.liquid`

```html
<script src="{{ 'embla-carousel.umd.js' | asset_url }}" defer="defer"></script>
<script src="{{ 'embla-carousel-accessibility.umd.js' | asset_url }}" defer="defer"></script>
<script src="{{ 'global.js' | asset_url }}" defer="defer"></script>
```

L'ordine **è essenziale**: quando `global.js` esegue `customElements.define()`, i costruttori vengono chiamati e `window.EmblaCarousel` deve già essere disponibile.

#### Struttura HTML richiesta da Embla (viewport > container > slides)

**Slideshow** (`<slideshow-component>`) — unico componente con Embla:
- Viewport: `.ms-slideshow__media-wrapper` (wrapper intermedio)
- Container: `<div id="Slider-xxx">` (figlio del wrapper)
- Slides: `.slideshow__slide`
- CSS viewport: `overflow: hidden`
- CSS container: `overflow: visible; scroll-snap-type: none;`

#### Inizializzazione JS

- `SlideshowComponent.initEmblaSlideshow()` → per slideshow (singolo item, full-width)
- `SliderComponent.initEmblaCarousel()` → disponibile ma **NON usato** (i caroselli generici usano scroll nativo)
- L'attributo runtime `data-ms-embla-active="true"` viene aggiunto **dopo** l'inizializzazione riuscita; se manca, Embla NON è attivo
- Retry automatico fino a 40 tentativi (ogni 100ms) se `window.EmblaCarousel` non è ancora disponibile

#### API Embla v9 (metodi usati nel codice)

| Metodo v9 | Descrizione | Vecchio nome v8 (NON usare) |
|---|---|---|
| `goToNext()` | Vai al prossimo snap | `scrollNext()` |
| `goToPrev()` | Vai al precedente snap | `scrollPrev()` |
| `goTo(index, instant?)` | Vai a un indice specifico | `scrollTo()` |
| `selectedSnap()` | Indice snap attuale | `selectedScrollSnap()` |
| `snapList()` | Lista posizioni snap | `scrollSnapList()` |
| `canGoToNext()` | Può andare avanti? | `canScrollNext()` |
| `canGoToPrev()` | Può andare indietro? | `canScrollPrev()` |

#### Opzioni Embla usate

```js
{
  loop: true/false,          // true per 'circular', false per 'rewind'/'none'
  align: 'start',
  container: this.slider,    // HTMLElement, passato esplicitamente
  slides: '[id^="Slide-"]',  // o '.slideshow__slide' per slideshow
  slidesToScroll: 'auto',    // 'auto' per page mode, 1 per single
  containScroll: 'trimSnaps' // solo quando loop=false
}
```

#### CSS overrides (in `ms_slideshow.css`)

**CRITICO**: usare `data-ms-embla="true"` (attributo statico dal Liquid) e NON `data-ms-embla-active="true"` (attributo runtime). Embla misura le slide durante l'inizializzazione: il layout CSS deve essere corretto PRIMA che Embla parta.

```css
slideshow-component[data-ms-embla="true"] .ms-slideshow__media-wrapper { overflow: hidden; }
slideshow-component[data-ms-embla="true"] .slideshow.slider--everywhere {
  overflow: visible; scroll-snap-type: none; scroll-behavior: auto;
  flex-wrap: nowrap;
}
slideshow-component[data-ms-embla="true"] .slideshow__slide {
  scroll-snap-align: none; flex: 0 0 100%; min-width: 0;
}
```

> **NON** creare override Embla per `slider-component` (caroselli generici). Il CSS grid Dawn (flex-wrap, column-gap, width proporzionali) è incompatibile con il `transform: translate3d()` di Embla.

#### Gestione loop con Embla

| Modalità | `loop` Embla | Comportamento navigazione |
|---|---|---|
| `circular` | `true` | Embla gestisce loop infinito nativamente |
| `rewind` | `false` | A fine/inizio, `goTo(0)` / `goTo(last)` manualmente |
| `none` | `false` | Bottoni disabilitati a fine/inizio, autoplay si ferma |

## 7.13 Dati Strutturati (JSON-LD) — Regole Obbligatorie

### 7.13.1 Limite 5000 caratteri per le descrizioni

> **Regola**: Tutti i campi `"description"` nei blocchi JSON-LD (`application/ld+json`) DEVONO essere troncati a **massimo 5000 caratteri**. Google ignora o penalizza le descrizioni che superano questo limite nei dati strutturati.

**Implementazione Liquid standard:**
```liquid
{%- assign sd_desc = source.description | strip_html | strip | truncate: 5000 -%}
{%- if sd_desc != blank -%}
"description": {{ sd_desc | json }},
{%- endif -%}
```

- Usare SEMPRE `strip_html` prima di `truncate` per rimuovere tag HTML
- Usare `strip` per eliminare spazi iniziali/finali dopo la pulizia
- Non usare il filtro built-in `product | structured_data` di Shopify: **non gestisce il limite** e può generare descrizioni illimitate. Usare invece lo snippet `ms_product-structured-data.liquid`

**Snippet di riferimento per prodotti**: `snippets/ms_product-structured-data.liquid`
- Genera JSON-LD Product schema completo (name, url, image[], description troncata, brand, sku, gtin, offers[])
- Da usare in tutte le sezioni che richiedono dati strutturati prodotto

**Checklist rapida:**

- [ ] Nessun uso di `product | structured_data` (usa `ms_product-structured-data`)?

## 7.14 Carousel Desktop — Pattern Obbligatorio per Slider dentro Container `page-width`

> **Problema**: Dawn's `slider-component` per default imposta `--desktop-margin-left-first-item: max(5rem, calc(...))`, pensato per slider che si estendono a tutta la larghezza della viewport. Quando il `<slider-component>` è **dentro un contenitore `page-width`** (caso normale nelle section MS), questo valore provoca:
> 1. **Primo elemento sfasato a destra** (margine-left iniziale non voluto)
> 2. **Scrollbar orizzontale e verticale** (larghezze degli item calcolate senza tener conto del peek)
> 3. **Anteprima prossimo elemento non funzionante** (il CSS default ignora `--ms-peek`)

### 7.14.1 Tre correzioni OBBLIGATORIE per desktop slider dentro page-width

#### Fix 1 — Variabile `is_full_width` nel blocco Liquid
```liquid
assign is_full_width = false
if sw == 'full_width'
  assign is_full_width = true
endif
```

#### Fix 2 — Blocco `{% style %}` dopo la sezione Liquid (solo quando `show_desktop_slider` e non full-width)
```liquid
{%- unless is_full_width -%}
  {%- if show_desktop_slider -%}
    {% style %}
      @media screen and (min-width: 990px) {
        #section-{{ section.id }} .slider.slider--desktop {
          margin-left: 0;
          scroll-padding-left: 0;
        }
        #section-{{ section.id }} .slider--desktop.grid--1-col-desktop .grid__item { width: calc((100% - var(--ms-peek, 4rem)) / 1 - var(--grid-desktop-horizontal-spacing)); }
        #section-{{ section.id }} .slider--desktop.grid--2-col-desktop .grid__item { width: calc((100% - var(--ms-peek, 4rem)) / 2 - var(--grid-desktop-horizontal-spacing)); }
        #section-{{ section.id }} .slider--desktop.grid--3-col-desktop .grid__item { width: calc((100% - var(--ms-peek, 4rem)) / 3 - var(--grid-desktop-horizontal-spacing)); }
        #section-{{ section.id }} .slider--desktop.grid--4-col-desktop .grid__item { width: calc((100% - var(--ms-peek, 4rem)) / 4 - var(--grid-desktop-horizontal-spacing)); }
        #section-{{ section.id }} .slider--desktop.grid--5-col-desktop .grid__item { width: calc((100% - var(--ms-peek, 4rem)) / 5 - var(--grid-desktop-horizontal-spacing)); }
        #section-{{ section.id }} .slider--desktop.grid--6-col-desktop .grid__item { width: calc((100% - var(--ms-peek, 4rem)) / 6 - var(--grid-desktop-horizontal-spacing)); }
      }
    {% endstyle %}
  {%- endif -%}
{%- endunless -%}
```
> Nota: l'uso di `{% style %}` (senza trattino) è **l'unica eccezione consentita** alle regole CSS inline, perché Shopify lo processa come CSS dinamico in `<head>` — non è inline nell'HTML. Questa è la tecnica ufficiale Shopify per CSS dipendente da dati di sezione.

#### Fix 3 — Attributo `style` su `<slider-component>`
```liquid
<slider-component
  ...
  {% unless is_full_width %}style="--desktop-margin-left-first-item: 0px"{% endunless %}
>
```

#### Fix 4 — Classe `slider` sempre presente sulla `<ul>` quando slider è attivo
```liquid
class="... {% if show_mobile_slider or show_desktop_slider %} slider{% if show_desktop_slider %} slider--desktop{% endif %}{% if show_mobile_slider %} slider--tablet grid--peek{% endif %}{% endif %}"
```
> La classe base `slider` deve esserci **sia per mobile che per desktop**. Senza di essa, il selettore `.slider.slider--desktop.contains-content-container .slider__slide` di Dawn non si applica e causa scrollbar verticali.

### 7.14.2 Formule peek proporzionali per colonna

Le formule nel `{% style %}` block usano `item_width = (100% - N*G) / (N + peek_fraction)`, dove:
- `none` (0 peek): `(100% - (N-1)*G) / N` → identico alla grid statica (vedi `base.css`)
- `small` (1/3 colonna): `(100% - N*G) / (N + 1/3)` = `3*(100% - N*G) / (3N+1)`
- `medium` (1/2 colonna): `(100% - N*G) / (N + 1/2)` = `2*(100% - N*G) / (2N+1)`
- `large` (3/4 colonna): `(100% - N*G) / (N + 3/4)` = `4*(100% - N*G) / (4N+3)`

Dove `G = var(--grid-desktop-horizontal-spacing)`.

> ⚠️ NON usare `var(--ms-peek)` con formula `(100% - --ms-peek) / N`: lascia sempre un gap residuo (pari a 1 gap column) visibile anche con `none`. Usare sempre le formule proporzionali per colonna sopra.

### 7.16 Gestione Testo Alternativo (ALT) Immagini

> **Principio**: ogni immagine caricata tramite `image_picker` nel live editor DEVE avere un testo ALT accessibile. Il sistema usa una **catena di fallback** a 4 livelli gestita dallo snippet `ms_image_alt.liquid`.

#### 7.16.1 Catena di Fallback ALT

| Priorità | Fonte | Descrizione |
|---|---|---|
| 1 | **Campo ALT personalizzato** | Testo inserito manualmente dall'utente nel live editor (setting `image_alt` di tipo `text`) |
| 2 | **ALT dell'immagine Shopify** | Testo ALT impostato nel media manager di Shopify (`image.alt`) |
| 3 | **Titolo del contesto** | Se l'immagine fa parte di un gruppo con un titolo (es. heading della sezione, titolo del blocco), viene usato quel titolo |
| 4 | **Nome file normalizzato** | Il nome del file viene estratto dall'URL, ripulito (rimozione estensione, sostituzione trattini/underscore con spazi) e con la prima lettera maiuscola |

#### 7.16.2 Snippet: `ms_image_alt.liquid`

Parametri:
- `image` (image object, obbligatorio): l'oggetto immagine Shopify
- `custom_alt` (string, opzionale): testo ALT dal setting personalizzato
- `fallback_title` (string, opzionale): titolo del contesto padre

**Utilizzo standard:**
```liquid
{%- capture img_alt -%}
  {%- render 'ms_image_alt', image: block.settings.image, custom_alt: block.settings.image_alt, fallback_title: block.settings.title -%}
{%- endcapture -%}
{%- assign img_alt = img_alt | strip -%}
```

Poi usare `img_alt` in:
```liquid
{{ image | image_url: width: image.width | image_tag: alt: img_alt }}
```

#### 7.16.3 Setting Schema Obbligatorio

Ogni `image_picker` nello schema DEVE essere seguito da un campo `text` per il testo ALT personalizzato:

```json
{
  "type": "image_picker",
  "id": "image",
  "label": "Immagine"
},
{
  "type": "text",
  "id": "image_alt",
  "label": "Testo alternativo (ALT)",
  "info": "Se vuoto, viene usato l'ALT dell'immagine caricata su Shopify. Se anche quello è vuoto, viene generato dal nome del file."
}
```

**Eccezioni** (NON aggiungere il campo ALT):
- Immagini di **sfondo** (`section_bg_image`): non sono contenuto, sono decorative → `aria-hidden="true"` o CSS `background-image`
- Immagini da **oggetti Shopify** (prodotti, collezioni, articoli): l'ALT è gestito dal media manager di Shopify
- **Poster video**: usano il campo `description` o il titolo del video come ALT

**Convenzioni per ID multipli**: se la sezione ha più immagini, usare suffissi:
- `image_alt` per l'immagine principale
- `image_2_alt` per la seconda immagine
- `cover_image_alt` per il poster video

#### 7.16.4 Parametro `fallback_title`

Quando il `fallback_title` viene passato al snippet, segue questa logica:

| Contesto | Valore `fallback_title` |
|---|---|
| Blocco con `title` | `block.settings.title` |
| Blocco con `heading` | `block.settings.heading` |
| Sezione con `heading` | `section.settings.heading` |
| Blocco multicolumn/multirow | `block.settings.title` o `block.settings.heading` |
| Nessun titolo disponibile | Omettere il parametro (il fallback sarà il nome file) |

## 8. Sistema di Temporizzazione (Scheduling)

> Sistema globale per controllare la visibilità degli elementi in base a data/ora.
> Composto da 3 file: variabile globale, snippet Liquid, utility JS.

### 8.1 Architettura

| File | Tipo | Scopo |
|---|---|---|
| `snippets/ms_datetime.liquid` | Variabile globale | Espone `window.MS_DATETIME` con timestamp server e utility JS. Da includere **una volta** in `layout/theme.liquid`. |
| `snippets/ms_schedule_check.liquid` | Snippet Liquid | Verifica server-side se un elemento è entro la finestra temporale. Output: `"true"` / `"false"`. |
| `assets/ms_schedule.js` | Utility JS | Verifica client-side gli elementi con `[data-ms-schedule]`. Fallback per pagine cachate dalla CDN. |

### 8.2 Come Aggiungere la Temporizzazione a un Elemento

**1. Settings nello schema** — aggiungere al block/section:
```json
{
  "type": "header",
  "content": "⏱ Temporizzazione"
},
{
  "type": "checkbox",
  "id": "schedule_enabled",
  "label": "Attiva temporizzazione",
  "default": false,
  "info": "Se attivo, l'elemento sarà visibile solo nella finestra temporale specificata."
},
{
  "type": "text",
  "id": "schedule_start",
  "label": "Data/ora inizio",
  "info": "Formato: YYYY-MM-DD HH:MM (es. 2025-12-25 09:00). Lascia vuoto per nessun limite."
},
{
  "type": "text",
  "id": "schedule_end",
  "label": "Data/ora fine",
  "info": "Formato: YYYY-MM-DD HH:MM (es. 2025-12-31 23:59). Lascia vuoto per nessun limite."
}
```

**2. Verifica Liquid (server-side)** — nel template:
```liquid
{%- capture is_visible -%}
  {%- render 'ms_schedule_check',
      schedule_enabled: block.settings.schedule_enabled,
      schedule_start: block.settings.schedule_start,
      schedule_end: block.settings.schedule_end
  -%}
{%- endcapture -%}
{%- assign is_visible = is_visible | strip -%}
{%- if is_visible == 'true' or request.design_mode -%}
  ... contenuto ...
{%- endif -%}
```

**3. Data attributes (client-side)** — per JS fallback:
```html
{%- if block.settings.schedule_enabled -%}
  data-ms-schedule
  data-ms-schedule-start="{{ block.settings.schedule_start }}"
  data-ms-schedule-end="{{ block.settings.schedule_end }}"
{%- endif -%}
```

**4. Indicatore Design Mode** — mostrare nell'editor Shopify:
```liquid
{%- if request.design_mode and is_visible != 'true' -%}
  <span style="opacity:0.6;font-size:0.75em;" aria-hidden="true">
    ⏱ Non visibile — fuori dalla finestra temporale
  </span>
{%- endif -%}
```

### 8.3 Sezioni con Temporizzazione Attiva

| Sezione | Elemento temporizzato |
|---|---|
| `sections/announcement-bar.liquid` | Ogni singolo messaggio/block |

> Aggiornare questa tabella ogni volta che si aggiunge la temporizzazione a una nuova sezione.

### 8.4 Note Importanti
- La verifica **server-side** (Liquid) è il meccanismo primario
- La verifica **client-side** (JS) è un fallback per pagine cachate dalla CDN Shopify
- In **Design Mode** (editor Shopify) mostrare SEMPRE tutti gli elementi con indicatore visivo
- Il formato data è **YYYY-MM-DD HH:MM** (es. `2025-12-25 09:00`)
- Campi vuoti = nessun limite (inizio o fine)

## 11. Funzionalità Globali da Rispettare

### 11.1 Modalità Solo Catalogo

Il tema ha una modalità "Solo catalogo" attivabile da **Impostazioni > Catalogo e Visibilita > Attiva modalita solo catalogo** (`settings.ms_catalog_mode`).

Quando attiva, una classe `ms-catalog-mode` viene aggiunta al `<body>` in `layout/theme.liquid`. Il CSS in `assets/ms_custom.css` nasconde automaticamente: icona carrello, cart drawer/notification, `product-form`, `.quick-add`, `.shopify-payment-button`, `.product-form__buttons`, `quantity-popover`, `.quantity`, `.product-form__quantity`, `.ms-min-order-label` (etichetta quantità minima d'ordine nelle card prodotto). La pagina carrello ridireziona alla home via JS.

**Nascondi prezzi** (`settings.ms_catalog_hide_prices`): checkbox subordinato alla modalità catalogo. Quando ENTRAMBI sono attivi, la classe `ms-catalog-hide-prices` viene aggiunta al `<body>` e il CSS nasconde `.price`, `price-per-item`, `.volume-pricing`. I dati strutturati (JSON-LD) continuano ad avere i prezzi per SEO.

**Regola**: se crei nuove sezioni/snippet con bottoni acquisto, form prodotto o riferimenti al carrello, assicurati che gli elementi usino le classi standard (`product-form`, `.product-form__buttons`, `.quick-add`, ecc.) oppure aggiungi regole CSS dedicate sotto `body.ms-catalog-mode` in `ms_custom.css`. Per le label di quantità minima d'ordine, usa la classe `ms-min-order-label` per garantire il nascondimento automatico in modalità catalogo.

### 11.2 Visualizzazione IVA sotto il Prezzo

Il tema mostra l'importo IVA calcolato sotto ogni prezzo, visibile **solo per Italia** (`localization.country.iso_code == 'IT'`). La logica è centralizzata in `snippets/price.liquid`.

**Settings** (gruppo "Catalogo e Visibilita"):
- `ms_show_iva` (checkbox) — attiva/disattiva
- `ms_iva_rate` (range, default 22) — aliquota % globale
- `ms_iva_prices_included` (checkbox) — se i prezzi includono già l'IVA
- `ms_iva_label` (text, default `+ IVA: -amount-`) — formato etichetta con segnaposto `-amount-`

**Aliquota per prodotto**: il metafield `custom.aliquota_iva` (numero decimale) sovrascrive l'aliquota globale.

**Regola**: se crei nuove sezioni/snippet che mostrano prezzi, usa sempre `{% render 'price', product: product %}` per ereditare automaticamente la riga IVA. Se per qualche motivo devi mostrare prezzi senza lo snippet `price.liquid`, replica la logica IVA o rendila visibile tramite lo stesso pattern (check `settings.ms_show_iva`, check paese IT, calcolo con `iva_rate`, label con `-amount-`).

### 11.3 Visibilità per Ospiti

Il tema permette di nascondere elementi specifici agli utenti non loggati, tramite 3 checkbox indipendenti in **Impostazioni > Catalogo e Visibilita > Visibilita per ospiti**.

| Setting | ID | Classe CSS su body | Elementi nascosti |
|---|---|---|---|
| Nascondi i prezzi | `ms_guest_hide_prices` | `ms-guest-hide-prices` | `.price`, `price-per-item`, `.volume-pricing` |
| Nascondi acquisto | `ms_guest_hide_buy` | `ms-guest-hide-buy` | Carrello, `product-form`, `.quick-add`, `.shopify-payment-button`, `.product-form__buttons`, `quantity-popover`, `.quantity`, `.product-form__quantity`, `.ms-min-order-label`, `.product__tax` |
| Nascondi varianti | `ms_guest_hide_variants` | `ms-guest-hide-variants` | `variant-selects` |

Le classi vengono aggiunte al `<body>` in `layout/theme.liquid` solo quando `customer` è nil (ospite) E il setting corrispondente è attivo. La pagina carrello ridireziona alla home via JS client-side per gli ospiti con `ms_guest_hide_buy` attivo.

**Priorità**: la modalità catalogo (sezione 11.1) ha **sempre** prevalenza. Se `ms_catalog_mode` è attivo, gli elementi sono nascosti per TUTTI (loggati e non), indipendentemente dai settings guest.

**Regola**: se crei nuove sezioni/snippet con prezzi, bottoni acquisto, form prodotto, varianti o riferimenti al carrello, assicurati che gli elementi usino le classi standard per garantire il nascondimento automatico sia in modalità catalogo che per ospiti. I selettori CSS guest sono identici a quelli della modalità catalogo.

## 12. Deploy Shopify via GitHub — Gestione Errori

Il tema è collegato a GitHub tramite l'integrazione nativa di Shopify (bidirezionale). **Quando Shopify rileva errori Liquid in uno qualsiasi dei file, rifiuta l'intero deploy** — nessun file viene aggiornato, e il tema resta alla versione precedente senza alcuna notifica esplicita.

**Sintomi di deploy fallito:**
- Le modifiche pushate su GitHub non appaiono sul sito live
- Shopify continua a pushare commit su GitHub (settings_data.json, ecc.) ma non tira le nostre modifiche
- Errori visibili tipo `Liquid error: Could not find asset snippets/xxx.liquid`

**Regole:**
1. **Se le modifiche non appaiono sul sito**, la prima cosa da verificare è che non ci siano errori Liquid nei file modificati. Shopify rifiuta silenziosamente il deploy.
2. **Mai pushare codice di debug** (tag `<p>` di test, `{%- comment -%}DEBUG...{%- endcomment -%}`) in produzione.
3. **Se il sync si interrompe**, andare su Shopify Admin → Online Store → Temi → "..." → Disconnetti e riconnetti GitHub.
4. **Dopo ogni modifica Liquid**, verificare mentalmente che tutti i tag siano bilanciati (`if/endif`, `for/endfor`, `comment/endcomment`, `unless/endunless`) e che non ci siano filtri o variabili inesistenti.
5. **Non inserire `{{ }}` dentro blocchi `{% comment %}`** — anche se tecnicamente ignorato, è una potenziale fonte di problemi.
